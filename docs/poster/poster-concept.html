Architecture description container

User submits checkout Frontend sends HTTP request to Orchestrator Orchestrator creates an order id and fans out gRPC init calls Transaction Verification, Fraud Detection, and Suggestions cache order data Verified orders enter the Order Queue Executor replicas elect one leader Leader dequeues the order and coordinates 2PC Payment and Books DB commit or abort together Books DB primary replicates committed stock to backups
Reader takeaway: validation and commitment are separate decisions. The orchestrator accepts valid orders; the executor leader commits irreversible side effects.

Causal verification container

Purpose: explain concurrency without a fragile sketch.

init a: items present -> c: card format b: user data -> d: user fraud join c + d -> e: card fraud -> f: suggestions

Vector clocks make the partial order visible: `a` and `b` can run in parallel, then `c` and `d` can overlap before event `e` joins both branches.

2PC container

Prepare: payment stages a charge; books DB checks stock and stages decrements.

Decision: all YES means Commit; any NO or timeout means Abort.

Known trade-off: 2PC can block after YES votes if the coordinator fails before publishing the decision.

Lower left

Key Insights

1

Causality is visible. Vector clocks show real concurrency, not just log order.

2

Side effects wait. Payment and stock changes happen only after Commit.

3

One leader acts. Bully election prevents executor replicas from processing the same queue item.

4

Retries are safe. Participants keep staged, committed, and aborted state per order.

Lower right

Challenges + Collaboration

  • 2PC blocking window: participants that voted YES cannot safely decide alone.
  • Async checkout: frontend receives `accepted`, then polls for `committed` or `aborted`.
  • Prototype durability: queue and status maps are in memory; Books DB replication is best effort.
